Decision guide

Choose the least powerful thing that comfortably works

Move up the stack only when the lower level becomes a real constraintвЂ”not because a larger toolchain is fashionable.

One page, modest interaction?

A few event listeners and DOM updates, with no complicated shared state.

Use one HTML file or HTML + CSS + JS.

Mainly need safer JavaScript?

Data shapes and refactoring are risky, but the interface is still straightforward.

Use vanilla TypeScript + a small build tool.

UI naturally component-shaped?

Repeated interactive units, shared state, conditional trees, and reuse justify components.

Use JSX/TSX with React or similar.

Mostly content?

Ship HTML by default and add client interactivity only to selected regions.

Use Astro or another content-first generator.

Response changes per request?

Private data, cookies, secrets, auth, or request-aware logic must run outside the browser.

Use a Worker, function, or server.

Does the host build?

A Git host can build source; a drag-and-drop static host expects browser-ready output.

Push source to a builder; upload dist/ to static hosting.

Four ways a URL becomes a website

The decisive question is whether the host serves files, runs a build, or executes code.

Static folder upload

Upload index.html and assetsвЂ”or a generated dist folder. No production build or server process runs.

Git-connected static build

The platform installs dependencies, runs a build command, then serves the configured output directory.

Edge/serverless function

The platform executes a request handler that returns HTML, JSON, redirects, or assets.

Traditional server

A long-running Node, Python, Ruby, PHP, or other application owns routing and responses.
